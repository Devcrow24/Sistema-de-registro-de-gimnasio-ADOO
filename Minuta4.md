## **Información y acuerdos tomados.**

---

**1.Revisión de los casos de uso ya escritos.**
En la reunión pasada ueron agragados la información mas basica de los casos de uso como puede ser:
- Actor primario (El actor que iniciaria el caso de uso)
- Personas interesadas y sus intereses (Personas involucradas y el impacto que generan al utilizar el caso de uso)
- Precondiciones (Las condiciones previas que deben tener los actores primarios para usar el caso de uso)
- Postcondiciones (El cambio esperado que tendrá el sistema despues de haber utilizado el caso de uso)
- Flujo basico (Los pasos basicos que trandrá el caso de uso para realizarse)
- Flujo alternativo (Los pasos extras por si algun paso del flujo basi cambia)
En esta reunión se escribieron las demás secciones para terminar de escribir por completo la especificación del caso de uso, para ello se escribiron las siguientes caracteristicas para aquellos casos de uso de prioridad muy alta:
- Alcance (Para el sistema al que se esta diseñando)
- Nivel (metas u objetivos del usuario)
- Requerimientos especiales (Requerimientos no funcionales) 
- Lista de variaciones y tecnologias de datos (Metodos I/O y formato de datos)

Además de eso pudimos tomar retroalimentación sobre nuestros requerimientos no funcionales que ya habiamos escrito, agregandoles otros requerimientos funcionales que eran necesarios para cumplir con la función de nuestros casos de uso, estos fueron:
- El sistema debe enviar una respuesta visual y auditiva (sonido corto o luz verde/roja) al cliente para indicar si el acceso fue aprobado o rechazado.
- El sistema debe poder conectarse con lector biométrico y lector de código, permitiendo actualizar o reemplazar los dispositivos sin modificar el software principal.
- El lector de huella debe reconocer al cliente en menos de 3 segundos el 95% del tiempo.
- La pantalla donde se muestra el mensaje de acceso debe ser visible y legible a 1 metro de distancia.
- El sistema debe permitir modo offline temporal para registrar entradas locales cuando falle la conexión al servidor central, sincronizando datos al restablecerse la conexión.

**3. Escribir los casos de uso de prioridad alta**
Los casos de uso de prioridad alta eran primordiales para el sistema y con ellos tambien identificamos mas requisitos no funcionales que era necesario que fueran especificados, los requisitos no funcionales que se agregaron pensando en los casos de uso extra que se escribieron fueron:
- Los botones y campos de acción más frecuentes deberán estar ubicados dentro del área visual principal del usuario.
- Se deberá incluir un mensaje de confirmación antes de finalizar una venta o cancelar una transacción.
- En caso de interrupción eléctrica o cierre inesperado, el sistema deberá guardar automáticamente la última transacción en curso
- El sistema debe registrar la fecha y hora de la renovación.
- El sistema debe generar un comprobante digital o impreso inmediatamente después del pago.
- El sistema deberá mostrar un mensaje de confirmación visual y sonoro al completar el pago exitosamente.
- El sistema debe mostrar los resultados de búsqueda en menos de 2 segundos el 90% del tiempo.
